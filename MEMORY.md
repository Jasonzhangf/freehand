# MEMORY

- 2026-06-15: Freehand architecture baseline uses `contracts + blocks + orchestrators` layering, with UI and provider semantics explicitly separated.
- 2026-06-15: Provider strategy is Rust-native first: OpenAI-compatible adapter may use `async-openai` or raw HTTP; Anthropic adapter defaults to raw HTTP/SSE until a stable Rust SDK is justified.
- 2026-06-15: Orchestrator crates are pure orchestration only. New helper or semantic functions must be searched for in existing libraries first and, when needed, added to `freehand-blocks` rather than `freehand-reason` or `freehand-node`.
- 2026-06-15: Freehand runtime home is `~/.freehand`; runtime logs, ledgers, replays, cache, and local state should converge there rather than leaking across ad hoc paths.
- 2026-06-15: Development and debugging must start from the function map owner and debug artifacts. If feature truth changes, update function map, architecture docs, project skill workflow, and durable memory in the same change set.
- 2026-06-15: Master/slave meaning is an input-permission configuration: whichever side is configured as master receives user input and dispatches tasks; slave in paired healthy mode only executes paired input and rejects unrelated direct input. Paired input source may be a user or another master.
- 2026-06-15: Each agent has a startup configuration file. Slave mode is chosen by startup config and currently confirmed to require at least listening interface, allowed pairing source IP, and pairing token.
- 2026-06-15: Feature work and bug fixes must both pass three self-checks before closure: information sufficiency, logic closure, and lifecycle completeness. If not satisfied, continue read-only tracing first and ask the user only when local truth cannot close the gap.
- 2026-06-15: Local multiple agents are managed by `config.toml`; one `config.toml` may define multiple local agents.
- 2026-06-15: `contracts.core` owns only cross-module shared semantic types plus shared IDs and error contracts. It excludes config schema, UI projection, and debug/trace envelope. Its types default to serializable, replayable, and persistable.
- 2026-06-15: `provider.semantic` v1 supports OpenAI-compatible and Anthropic. It requires both single-shot and stream interfaces, unified semantic outputs, debug-only raw event retention, explicit capability declaration, and recovery classification with periodic windows expressed in seconds and defaulting to half-hour, five-hour, or daily-midnight windows when provider hints are absent.
- 2026-06-15: `reason.turn` writes truth per turn, projects conversation view from turns, and is the only writer of session truth. Freehand does not stop merely because provider finish_reason is stop/end_turn; stopping requires valid completion schema evaluation.
- 2026-06-15: `node.master-slave` v1 is local one-master one-slave only, pairs via WebSocket handshake, uses fixed pairing source per slave config, and keeps slave listening after pairing loss for re-pairing. Master must be able to delegate work, query progress, directly talk, and subscribe to slave turn output.
- 2026-06-15: `ui.protocol` v1 supports CLI and WebUI, exposes full command/projection/query/subscribe truth, keeps query and subscribe separate, carries explicit source identity fields, shows only terminal projected text, and renders slave turn as WebUI-only separate sub-stream card.
- 2026-06-15: Config core v1 is fixed so far: only `~/.freehand/config.toml`; multi-agent layout uses `[agents.<name>]`; one process starts one agent by CLI-selected name; config changes are restart-only; `allowed_pair_ip` is optional and `pair_token` is an environment-variable reference.
- 2026-06-15: Freehand test policy is three-layered by default: module white-box, module black-box, and project black-box. Each feature map entry must declare all three required test groups, and compile/regression flow must run them under the workspace test umbrella plus architecture gates.
- 2026-06-15: Freehand splits feature ownership from code-call truth. `docs/architecture/feature-map.md` owns feature owner and validation routing; `docs/function-maps/` owns request/response/error mainlines, shared multi-reference function descriptions, and code-bound call tables. No feature implementation is closed unless the function-map binding stays in sync with code.
- 2026-06-15: `config.core` baseline is now implemented. `freehand-config` loads only `~/.freehand/config.toml`, validates `[agents.<name>]`, requires `name`, `mode`, and `pair_token`, enforces `name == table key`, and resolves `pair_token` as an environment-variable reference during selected-agent startup. `freehand-cli --agent <name>` is the current project black-box startup harness for this module.
- 2026-06-15: `contracts.core` baseline is now implemented. `freehand-contracts` provides shared ID newtypes, first-pass request nodes, semantic response event types, tool call/result contracts, token usage, terminal status, classified error contracts, and a shared request validator. These contracts are JSON-serializable/replayable/persistable by default and now serve as the first real cross-module semantic truth.
- 2026-06-15: `provider.semantic` baseline now explicitly supports OpenAI `responses` plus Anthropic Messages-family semantics at the shared semantic layer. `freehand-provider-core` now owns provider family/protocol descriptors, raw-retention policy, unified semantic request building, normalized adapter-event mapping, and provider error classification. Provider protocol reference snapshots are stored under `docs/references/provider-protocols/`, with a local `provider-protocols` skill for implementation-time lookup.
- 2026-06-15: shared tool-call and usage contracts were tightened for provider work. `ToolArgument` now preserves structured JSON values instead of string-only payloads, and `TokenUsage` now carries optional `total_tokens`, `reasoning_tokens`, and `finish_reason` while keeping explicit cache counters for replay-safe accounting.
- 2026-06-15: `contracts.core` now carries typed context-planning request truth. `ReasonReq02ContextComposedInput` uses `context_segments` instead of ad hoc context pairs, `ReasonReq03ProviderPayload` uses `input_segments` instead of `rendered_input`, and shared contracts now define `ContextSegment`, kind/stability/cache-policy/role enums, provenance, and request validators for both composed input and provider payload boundaries.
- 2026-06-15: provider adapter baseline is now implemented. `freehand-provider-openai` supports OpenAI `responses` and `chat completions`; `freehand-provider-anthropic` supports Anthropic Messages. Both adapters now render semantic requests into protocol-specific request bodies, parse single-shot and stream payloads into shared semantic outputs, accumulate partial tool-call/tool-use JSON until complete, and keep wire DTOs private to adapter crates. Live HTTP execution remains intentionally out of scope for this baseline.
- 2026-06-15: provider adapters now consume typed input segments and render them with shared segment rendering logic from `freehand-blocks`; provider adapters no longer depend on a single mixed rendered prompt string contract.
- 2026-06-15: `reason.turn` has not yet fully aligned with the referenced `../Deepseek-reasonix` context orchestration. Before changing the real context planner, inspect reasonix and document cache-relevant ordering. Target direction is stable/volatile typed context segments, stable segment reuse across turns, and debug-visible cache boundaries.
- 2026-06-15: Reason and provider modules must remain implementation-independent. `freehand-reason` owns context orchestration, turn truth, completion control, and broadcast; provider adapters own protocol rendering/parsing only. `freehand-reason` must not depend on provider adapter crates, and provider adapters must not depend on `freehand-reason`.
- 2026-06-15: Cross-module metadata and request data require hard isolation. Metadata/debug/provider/cache fields must use separate types and builders from request-chain content fields. Metadata must not be smuggled into prompt/request text, and request content must not be recovered from metadata/debug fields.
- 2026-06-15: `reason.turn` baseline is now implemented. `freehand-reason` owns per-turn truth, provider-output materialization, tool-result re-entry ownership, session projection from turns, and non-blocking semantic broadcast. Completion-schema validation lives in `freehand-blocks`; provider finish reasons do not stop turns by themselves. Valid complete and blocked submissions terminalize the turn, invalid submissions are rejected, and `next_step` submissions remain non-terminal.
- 2026-06-15: `node.master-slave` baseline is now implemented as a local runtime semantic layer in `freehand-node`. `LocalNodeRuntime` validates master/slave bootstrap fields, models websocket pairing intent, materializes paired/rejected/listening node status, restricts slave input to the active paired source, preserves relisten behavior after pairing loss, stores delegated-task progress through `freehand-ui-protocol`, and publishes slave turn projections for subscribed master/UI consumers. Real websocket IO wiring and final wire schema remain intentionally out of scope for this baseline.
- 2026-06-15: `reason.context-planner` design is now locked. Freehand will combine Reasonix cache-first session discipline with Codex typed-fragment discipline: stable prefix remains stable across ordinary turns, context rewrites happen only through explicit rewrite gates, metadata/request payloads stay hard-isolated, and parent context is enriched from subagent work only through a typed final-conclusion segment rather than raw child transcript injection.
- 2026-06-15: `reason.context-planner` baseline is now implemented in `freehand-blocks`. `plan_context` owns typed segment admission, stable-before-volatile ordering, user-turn segment ownership, token-budget rejection, raw-subagent-transcript rejection by provenance, and cache diagnostics generation. `freehand-reason` now delegates turn-start request planning to this planner and keeps planner diagnostics outside request content. Rewrite mode/version sourcing and explicit compaction/rollback/resume gates are now closed by `reason.session-history`; remaining planner gap is runtime tool-schema fingerprint wiring.
- 2026-06-15: `reason.session-history` baseline is now implemented in `freehand-reason`. `SessionHistory` owns session base context, session-owned `rewrite_version`, one-shot `current_rewrite_mode`, rewrite ledger evidence, and persisted JSON/file snapshots. `ReasonTurnEngine::start_turn` now reads rewrite mode/version from `SessionHistory` and clears non-ordinary rewrite mode only after successful turn startup. Explicit `stage_compaction`, `stage_rollback`, and `stage_resume_rebuild` methods are the only baseline rewrite-version bump paths.
- 2026-06-15: `reason.persistence` design is now locked. Freehand reason persistence is split into three layers: authoritative snapshots under `~/.freehand/state/turns`, append-only reason/provider ledgers under `~/.freehand/ledgers`, and derived UI/index sidecars under `~/.freehand/state/ui` plus `~/.freehand/cache/session-index`. Only `freehand-reason` may write authoritative session/turn truth; provider raw ledgers and UI sidecars are never recovery truth.
- 2026-06-15: `reason.rewrite-policy` baseline is now implemented in `freehand-blocks`. Compaction trigger policy follows Reasonix evidence for soft notice (`50%`), auto compaction (`80%`), force compaction (`90%`), tail cap (`min(window * 0.5, 16384)`), and auto-pause after two ineffective compactions. Freehand-specific rollback and resume-rebuild trigger policy is now explicit because the reference project does not expose equivalent persisted session-history gates. `ReasonRewriteRuntime` in `freehand-reason` is the single baseline consumer that applies policy decisions before calling `SessionHistory::stage_*`. Provider usage enters rewrite policy only through `prompt_tokens_from_usage`, which maps `TokenUsage.input_tokens` to prompt pressure and rejects invalid usage. `freehand-testkit::ReasonRuntimeHarness` now provides project black-box coverage from provider semantic output to turn truth to usage-driven rewrite policy. Production CLI/server runtime loop wiring with real provider usage events and persisted recovery payloads remains pending.
- 2026-06-15: `app.cli-runtime-smoke` baseline is now implemented in `apps/freehand-cli`. The binary now supports `reason-e2e --agent <name> --scenario <usage-compaction|recovery-block>`, which loads real config from `~/.freehand/config.toml`, selects one agent, and routes through the shared `freehand-testkit::ReasonRuntimeHarness` instead of duplicating runtime semantics in the app layer. This is the current app-boundary end-to-end smoke path and is covered by `cargo test -p freehand-cli`.
- 2026-06-15: `config.core` now owns the provider registry as well as the agent registry. `~/.freehand/config.toml` contains `[agents.<name>]` plus `[providers.<id>]`; each agent binds exactly one provider id, and `LoadedConfig::select_agent` is now the single runtime-selection path for pair token, provider protocol, and provider auth resolution. Provider protocol must be explicit in config and must not be guessed from provider type or model name. CLI/config projections must never print resolved provider secrets.
- 2026-06-15: `config.core` provider config must only contain fields with current owner and runtime meaning. Unused knobs such as `transport_backend` must not enter config truth, and provider entries reject unknown fields so unimplemented settings fail explicitly instead of being silently accepted.
- 2026-06-15: `provider.anthropic-adapter` now has live `minimonth` replay fixtures for Anthropic-compatible Messages API single-shot and SSE stream responses. Fixtures live under `crates/freehand-provider-anthropic/fixtures/` and lock thinking/text/usage/cache/terminal mapping through `AnthropicAdapter::parse_response` and `AnthropicAdapter::parse_stream_event`.
- 2026-06-15: `provider.anthropic-adapter` now has an HTTP executor baseline in `AnthropicExecutor`. It owns Anthropic-compatible HTTP POST wiring, required auth/version/content headers, non-success HTTP status errors, single-shot parsing through `AnthropicAdapter::parse_response`, and collected-SSE parsing through `AnthropicAdapter::parse_stream_event`. Default regression uses local mock servers, not live credentials.
- 2026-06-15: `provider.reason-live-bridge` is now implemented in `freehand-testkit` as the first config-selected live path into `reason.turn`. It bridges `SelectedAgentConfig` into provider descriptor + Anthropic executor config, starts one turn, builds provider semantic request from turn-owned payload, executes single-shot or collected-SSE Anthropic requests, and applies provider-neutral outputs back into turn truth while capturing broadcasts. `freehand-cli reason-live` reuses this path; `freehand-reason` still does not depend on provider adapter crates.
- 2026-06-15: Anthropic live stream execution is no longer collected-SSE only. `AnthropicExecutor::execute_stream_with` now reads SSE event boundaries incrementally from the HTTP response, parses each semantic batch immediately, and `provider.reason-live-bridge` applies those batches into `ReasonTurnEngine` before stream completion. Regression tests explicitly lock both halves: executor callback-before-release and bridge broadcast-before-release.
- 2026-06-15: live reason completion is now governed by a tagged JSON loop. Freehand requires `<freehand_completion>...</freehand_completion>` in model-visible text, parses it through `freehand-blocks`, strips the tagged block from visible text projection, rejects invalid schema with field-level feedback, treats `claim=continue` as the next execution signal, and fails the live request after 3 invalid schema retries. `freehand-reason` owns the failed terminal write; `freehand-testkit` owns the multi-round composition loop.
